# Personal AI Employee — Gold Tier

An autonomous AI agent that monitors your Gmail and LinkedIn, drafts responses, posts to social media, manages your Odoo CRM, and executes approved actions — all without you touching the apps. Built on Claude Code + Obsidian as the shared brain.

```
Gmail / LinkedIn / Files / Cron
         ↓
   Watchers (Python, PM2)
         ↓
   vault/Needs_Action/
         ↓
   Planning Engine (Claude Code)
         ↓
   vault/Pending_Approval/
         ↓
   You review → move to vault/Approved/
         ↓
   Approval Executor (dispatches to skills)
         ↓
   Email · LinkedIn · Twitter · Facebook · Instagram · Odoo
         ↓
   vault/Done/ + JSONL audit log
```

---

## What It Does

| Capability | Detail |
|---|---|
| Gmail monitoring | Polls Gmail every 2 min, detects unread emails, drafts replies |
| File drop | Drop any file in `vault/Inbox/` → processed automatically |
| LinkedIn posting | Posts to your company page via the official LinkedIn API |
| Twitter/X posting | Posts to X via Playwright (persistent Chromium session) |
| Facebook posting | Posts to your Facebook Page via the Graph API |
| Instagram posting | Posts photo + caption to Instagram via the Content Publishing API |
| Odoo CRM | Creates leads, logs activities, and drafts invoices (never auto-posts) |
| CEO Briefing | Weekly audit of Done/, Logs/, Plans/ → structured executive report |
| Human-in-the-loop | Nothing is sent without you approving it first |
| Audit trail | Every action logged to `vault/Logs/YYYY-MM-DD.jsonl` |
| Ralph Wiggum loop | Stop hook re-injects task prompt until vault is fully processed |
| Error recovery | Failed actions go to `vault/Failed/`; retry with `--retry-failed` |

## Setup Guide

This section walks you through running this agent on your own machine from scratch.

### Prerequisites

| Requirement | Version | Purpose |
|---|---|---|
| [Claude Code](https://claude.ai/code) | Latest | Reasoning engine |
| Python | 3.11+ | Watchers and skill scripts |
| Node.js | v18+ | PM2 process manager |
| [Obsidian](https://obsidian.md) | Any (free) | Vault GUI (optional but recommended) |
| A Gmail account | — | Gmail watcher + sender |

---

### Step 1 — Clone the repository

```bash
git clone https://github.com/your-username/full_time_employee.git
cd full_time_employee
```

---

### Step 2 — Python environment

```bash
python3 -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate

pip install -r watchers/requirements.txt
pip install pyyaml python-dotenv playwright playwright-stealth requests
playwright install chromium
```

### Step 3 — Install PM2

PM2 keeps all background processes alive and restarts them on crash or reboot.

```bash
npm install pm2
```

Optional — make `pm2` available system-wide:

```bash
npm install -g pm2
```

Or use the local binary:

```bash
ln -s "$(pwd)/node_modules/.bin/pm2" ~/.local/bin/pm2
```

### Step 4 — Create your `.env` file

```bash
cp .env.example .env
```

Then edit `.env` with your values. The sections below explain where each value comes from.

```env
# ── Twitter/X ─────────────────────────────────────────────────────

# ── Facebook ──────────────────────────────────────────────────────

FACEBOOK_PAGE_ACCESS_TOKEN=EAAxxxxx...

# ── Instagram ─────────────────────────────────────────────────────
FACEBOOK_APP_ID=xxxxx
FACEBOOK_APP_SECRET=xxxxx
FACEBOOK_ACCESS_TOKEN=EAAxxxxx    # user token with instagram_content_publish scope
INSTAGRAM_USER_ID=123456789       # optional — auto-detected on first run

# ── Odoo (optional) ───────────────────────────────────────────────
ODOO_URL=http://localhost:8069
ODOO_DB=your_db_name
ODOO_USERNAME=admin
ODOO_PASSWORD=your_password
```

### Step 5 — Gmail API credentials

1. Go to [Google Cloud Console](https://console.cloud.google.com)
2. Create a project → enable the **Gmail API**
3. Go to **APIs & Services → Credentials → Create Credentials → OAuth 2.0 Client ID**
4. Application type: **Desktop app** → download the JSON file
5. Save it somewhere safe and set `GMAIL_CREDENTIALS` in `.env` to its absolute path

---

### Step 6 — Gmail OAuth (one-time login)

The first time the Gmail watcher runs it opens a browser for consent:

```bash
source .venv/bin/activate
python watchers/gmail_watcher.py vault
```

Authorise access in the browser — grant **both** `Gmail Read` and `Gmail Send` scopes when prompted. The token is saved to `watchers/.state/gmail_token.json` and reused automatically from that point forward.

> If you later see a `gmail.send` permission error, delete `watchers/.state/gmail_token.json`, restart `watcher-gmail`, and re-authorise.

---

### Step 7 — LinkedIn (official API — for company page posts)

This uses LinkedIn's official UGC Posts API, not browser automation.

**Prerequisites:**
1. You must have a **LinkedIn Company Page** where you are an Administrator
2. Create a LinkedIn Developer App at [linkedin.com/developers](https://www.linkedin.com/developers/)
3. Add the product **"Share on LinkedIn"** to your app (grants `w_organization_social` scope)
4. Set the redirect URL to: `http://localhost:8765/`
5. Copy your **Client ID** and **Client Secret** into `.env`

**Run the auth helper:**

```bash
source .venv/bin/activate
python watchers/auth_linkedin_api.py
```

A browser window opens for the OAuth consent flow. After authorising, the script prints your `LINKEDIN_API_ACCESS_TOKEN` and `LINKEDIN_API_ORG_URN` — paste both into `.env`.

> LinkedIn access tokens last ~60 days. Re-run `auth_linkedin_api.py` when the token expires.

---

### Step 8 — Twitter/X (one-time login)

Twitter uses a Playwright persistent session. The login is a **3-step flow** — read the prompts carefully.

```bash
source .venv/bin/activate
python watchers/auth_twitter.py
```

A browser window opens. Complete all three steps:

1. Enter your email or phone number → click **Next**
2. Enter your **@username** (identity verification, not password) → click **Next**
3. Enter your **password** → click **Log in**

Wait for your home feed to load, then **close the browser**. The session is saved to the path set in `TWITTER_SESSION_PATH`.

> Do NOT use "Continue with Google" — it gets blocked by X's automation detection.

---

### Step 9 — Facebook (Graph API token)

Facebook uses the official Graph API — no browser automation needed.

1. Go to [developers.facebook.com](https://developers.facebook.com) → **My Apps → Create App** (Business type)
2. Add products: **Facebook Login** and **Pages API**
3. Open [Graph API Explorer](https://developers.facebook.com/tools/explorer/)
4. Select your app → click **Generate Access Token**
5. Grant scopes: `pages_show_list`, `pages_manage_posts`, `pages_read_engagement`
6. Call `GET /me/accounts` — find your Page in the response
7. Copy the `access_token` (Page Access Token) and `id` (Page ID)

Convert to a long-lived token (~60 days):

```
GET https://graph.facebook.com/v21.0/oauth/access_token
    ?grant_type=fb_exchange_token
    &client_id={APP_ID}
    &client_secret={APP_SECRET}
    &fb_exchange_token={SHORT_LIVED_TOKEN}
```

Add to `.env`:

```env
FACEBOOK_PAGE_ID=123456789
FACEBOOK_PAGE_ACCESS_TOKEN=EAAxxxxx...
```

> Page access tokens expire every ~60 days. Refresh by generating a new user token in Graph API Explorer and calling `/me/accounts` again.

---

### Step 10 — Instagram (Content Publishing API)

Instagram requires your account to be a **Business** or **Creator** account linked to a Facebook Page.

1. Your Facebook Developer App (from Step 9) also handles Instagram
2. In Graph API Explorer, generate a token with these additional scopes:
   - `instagram_basic`
   - `instagram_content_publish`
3. Add to `.env`:

```env
FACEBOOK_APP_ID=xxxxx
FACEBOOK_APP_SECRET=xxxxx
FACEBOOK_ACCESS_TOKEN=EAAxxxxx    # the user token with instagram_content_publish
FACEBOOK_PAGE_ID=123456789
```

The script auto-detects your `INSTAGRAM_USER_ID` from your page on first run. You can also pin it manually:

```bash
curl "https://graph.facebook.com/v21.0/{PAGE_ID}?fields=instagram_business_account&access_token={TOKEN}"
```

Copy the `id` value and add `INSTAGRAM_USER_ID=<id>` to `.env`.

> Instagram posts **require a publicly accessible HTTPS image URL** — local file paths are not accepted by the API. Host your image on S3, Cloudflare Images, Imgur, or any CDN before posting.

### Step 11 — Odoo (optional)

If you want CRM and accounting integration:

1. Install [Odoo Community Edition](https://www.odoo.com/page/download) locally (or use a cloud instance)
2. Create a database and an admin user
3. Enable the **CRM** and **Accounting** modules
4. Add to `.env`:

```env

ODOO_DB=your_database_name
ODOO_USERNAME=admin
ODOO_PASSWORD=your_odoo_password
```

The Odoo skill creates leads and logs activities automatically, and can create **draft invoices** — but never posts them without your explicit approval.

## Daily Workflow

### Automatic (nothing to do)

1. An email arrives → Gmail watcher detects it within ~2 minutes
2. Planning engine reads the email, drafts a reply, writes an approval request to `vault/Pending_Approval/`
3. The file sits there until you review it

### Your only job — approve or reject

Open `vault/Pending_Approval/` in Obsidian (or any file manager):

- **Approve:** move the file to `vault/Approved/` → executor sends it within 30 seconds
- **Reject:** move the file to `vault/Rejected/` → no action taken

## Sending Manually (Without a Trigger)

You can create approval files directly — no incoming message needed. Save the file to `vault/Approved/` and the executor picks it up within 30 seconds.

> **The YAML frontmatter block at the top is required.** Without it the executor skips the file.

### Send an email

```markdown
---
type: approval_request
action: send_email
status: approved
---

- **Target:** recipient@example.com
- **Subject / Title:** Your subject line here

  Hi,

  Your email body here.

  Best regards,
  Your Name
```

## Triggering the CEO Briefing

Drop a file named `BRIEFING_DAILY_YYYY-MM-DD.md` (any content) into `vault/Inbox/`. The filesystem watcher detects it and triggers the `ceo-briefing` skill, which reads your `Done/`, `Logs/`, `Plans/`, and `Dashboard.md` to generate a structured briefing in `vault/Briefings/`.

You can automate this with a cron job:

```bash
# Generate a briefing trigger every Monday at 8:00 AM
0 8 * * 1 echo "# Briefing trigger" > /path/to/full_time_employee/vault/Inbox/BRIEFING_DAILY_$(date +\%F).md
```

## Configuration

### Autonomy rules

Edit `vault/Company_Handbook.md` to define what Claude can do without approval. The agent reads this file before every vault-operator session.

### Ralph Wiggum (autonomous mode)

The Stop hook only activates when Claude is invoked in autonomous mode (via cron or `scripts/autonomous.sh`). It never fires during interactive sessions. To run Claude autonomously on a folder of tasks:

```bash
bash scripts/autonomous.sh
```

This creates `.claude/hooks/.autonomous_mode` before invoking `claude -p`, and removes it when the vault is clear.

### Poll intervals

- `approval-executor`: edit the `--interval` arg in `ecosystem.config.js` (default: 30s)
- `watcher-gmail`: edit `check_interval` in `watchers/gmail_watcher.py` (default: 120s)

Restart after any change: `pm2 restart all`

## Troubleshooting

| Problem | Fix |
|---|---|
| Nothing appearing in `Needs_Action/` | Check `pm2 logs watcher-gmail` — likely a credentials or token scope issue |
| Executor skips approval file | Confirm the file has a `---` YAML frontmatter block with `action:` and `status: approved` |
| Gmail `invalid_scope` error | Delete `watchers/.state/gmail_token.json`, restart `watcher-gmail`, re-authorise in browser |
| LinkedIn post fails with 401 | Token expired (~60 days) — re-run `python watchers/auth_linkedin_api.py` |
| Twitter post fails / session expired | Re-run `python watchers/auth_twitter.py` and complete the 3-step login |
| Facebook/Instagram token expired | Regenerate in Graph API Explorer, call `/me/accounts`, update `.env` |
| Action moved to `vault/Failed/` | Check `pm2 logs approval-executor`. Fix the issue, then retry: `python .claude/skills/approval-executor/scripts/execute.py --vault ./vault --retry-failed` |
| PM2 process keeps restarting | `pm2 logs <process-name>` — usually a missing `.env` var or wrong file path |
| Instagram: image URL rejected | URL must be public HTTPS — local paths are not accepted by the API |
| Odoo connection refused | Confirm Odoo is running on `ODOO_URL` and the DB name matches `ODOO_DB` |

## Security

- Never commit `.env` — it is already in `.gitignore`
- Session data (`.sessions/`) is excluded from git
- Gmail token (`watchers/.state/`) is excluded from git
- All outbound actions require human approval — nothing is sent autonomously
- Odoo draft invoices are never posted without explicit approval
- Every action is logged to `vault/Logs/YYYY-MM-DD.jsonl` for audit purposes
- Rate limiting: max 10 actions per hour (enforced by the executor)
