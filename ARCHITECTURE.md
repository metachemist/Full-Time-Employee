# AI Employee — Architecture & Lessons Learned

> Gold Tier implementation — `gold-tier` branch

---

## 1. System Overview

The AI Employee is a **locally-running autonomous agent** built on Claude Code as the reasoning engine. It monitors external inputs (email, LinkedIn, file drops), decides what to do, routes sensitive actions through human approval, and executes approved actions via dedicated skill scripts.

```
External Sources
   (Gmail · LinkedIn · File drops · Cron)
         │
         ▼
  Watchers (Python)
  ─────────────────────────────────────────
  gmail_watcher.py          polls Gmail API every 2 min
  linkedin_watcher.py       polls LinkedIn via Playwright every 5 min
  filesystem_watcher.py     event-driven (watchdog) on vault/Inbox/
         │
         ▼ writes .md files
  vault/Needs_Action/
         │
         ▼ reads + plans
  Planning Engine (Claude Code + vault-operator skill)
         │
         ├─── Auto-approved ────────────────────────────▶ Execute directly
         │                                                  ↓
         └─── Needs approval ──▶ vault/Pending_Approval/  vault/Done/
                                        │
                                 human moves file
                                        ▼
                               vault/Approved/
                                        │
                                        ▼
                          Approval Executor (execute.py)
                          ─────────────────────────────
                          dispatches to skill scripts →
                          Gmail · LinkedIn · Twitter · Facebook
                          Instagram · Odoo
                                        │
                                        ▼
                                  vault/Done/
                                  vault/Logs/ (JSONL audit)
```

## 2. Vault as Shared State (Obsidian)

The Obsidian vault is the **GUI and long-term memory**. Every component reads and writes markdown files — this means:

- A human can inspect any step by opening the vault in Obsidian
- Claude can resume work after restarts by reading `Dashboard.md` and `Needs_Action/`
- The audit trail lives in `vault/Logs/YYYY-MM-DD.jsonl`

### Vault Layout

```
vault/
  Dashboard.md            Live status — updated after every session
  Company_Handbook.md     Autonomy rules — what needs approval vs. auto
  Inbox/                  Drop folder — filesystem_watcher watches this
  Needs_Action/           Watcher output — Claude reads and processes
  Plans/                  PLAN_*.md — created before multi-step tasks
  Pending_Approval/       APPROVAL_*.md — awaiting human decision
  Approved/               Human-moved files — executor processes these
  Rejected/               Human-rejected items — archived
  Done/                   Completed items — never deleted
  Failed/                 Permanently failed actions — for review
  Briefings/              CEO briefings — auto-generated by cron
  Logs/                   Append-only JSONL audit (one file per day)
```

### File Naming Conventions

| Type | Pattern | Location |
|------|---------|----------|
| Action item | `TYPE_description_YYYY-MM-DD_HHMMSS.md` | `Needs_Action/` |
| Plan | `PLAN_<desc>_YYYY-MM-DD.md` | `Plans/` |
| Approval request | `APPROVAL_ACTION_<desc>_YYYY-MM-DD.md` | `Pending_Approval/` |
| CEO briefing | `BRIEFING_<SCOPE>_YYYY-MM-DD.md` | `Briefings/` |
| Audit log | `YYYY-MM-DD.jsonl` | `Logs/` |

## 3. Component Architecture

### 3.1 Watchers (Perception Layer)

`watchers/base_watcher.py` — abstract base class:
- `_write_audit()` — appends JSONL to `vault/Logs/`
- `@with_retry` decorator — exponential backoff for flaky network calls
- Lifecycle events logged: `watcher_started`, `item_created`, `watcher_error`

| Watcher | Trigger | Creates |
|---------|---------|---------|
| `filesystem_watcher.py` | New file in `vault/Inbox/` | `FILE_*` or `BRIEFING_*` action |
| `gmail_watcher.py` | New Gmail (unread) | `EMAIL_*` action |
| `linkedin_watcher.py` | New connection/DM | `LINKEDIN_LEAD_*` action |

Typed dispatch in `filesystem_watcher.py`:
- `BRIEFING_*.md` → `type: briefing_request` (auto-approved)
- Everything else → `type: file_drop` (generic)

### 3.2 Planning Engine (Reasoning Layer)

Claude Code with the `vault-operator` skill reads `Needs_Action/` and:
1. Reads `Company_Handbook.md` for autonomy rules
2. Classifies each item: auto-approved or needs human sign-off
3. For auto-approved: creates Plan → executes → moves to Done
4. For approval-needed: writes `APPROVAL_*.md` to `Pending_Approval/`

**Human-in-the-Loop**: The human moves the approval file from `Pending_Approval/` to `Approved/`. The approval executor picks it up within 30 seconds.

### 3.3 Approval Executor (`execute.py`)

Polls `vault/Approved/` every 30 seconds (PM2 daemon). For each file:
1. Parses YAML frontmatter: `action:`, `status:`
2. Extracts body fields: `Target:`, `Subject / Title:`, `Message / Content`
3. Dispatches to the appropriate skill script with built CLI args
4. On success: moves to `Done/`, writes audit log
5. On failure/timeout: moves to `Failed/`, writes error audit log

**Retry mechanism**: `--retry-failed` flag moves `Failed/` items back to `Approved/` with `status: approved` reset.

**Rate limiting**: Max 10 actions/hour (in-memory, resets on restart).

### 3.4 Skills (Action Layer)

Each skill is a self-contained folder under `.claude/skills/`:

```
.claude/skills/
  approval-executor/    Polls vault/Approved/ and dispatches
  browsing-with-playwright/  MCP server for browser automation
  ceo-briefing/         Reads vault state → structured exec report
  facebook-poster/      Playwright → Facebook text post
  gmail-sender/         Gmail API → sends approved email
  instagram-poster/     Playwright → Instagram photo+caption post
  linkedin-dm/          Playwright → LinkedIn DM
  linkedin-poster/      Playwright → LinkedIn text post
  odoo-crm/             JSON-RPC → Odoo CRM leads + draft invoices
  twitter-poster/       Playwright → X/Twitter post
  vault-operator/       Claude skill for reading/writing the vault
```

Each skill has:
- `SKILL.md` — description, instructions, approval file format
- `scripts/` — executable Python script(s)

### 3.5 Ralph Wiggum Loop (Autonomy Layer)

The Stop hook (`.claude/hooks/ralph_wiggum.py`) intercepts Claude's exit:
- Only fires when `.claude/hooks/.autonomous_mode` flag file exists
- Checks `vault/Needs_Action/` and `vault/Approved/` for pending work
- Re-injects the vault-operator prompt if work remains
- Removes the flag when work is complete (clean exit)

`scripts/autonomous.sh` sets the flag before invoking `claude -p "..."` (used by cron). Interactive sessions are never interrupted.

## 5. PM2 Process Map

```
ecosystem.config.js
├── watcher-filesystem       vault/Inbox/ → Needs_Action
├── watcher-gmail            Gmail API → Needs_Action
├── watcher-linkedin         LinkedIn → Needs_Action
├── approval-executor        vault/Approved/ → actions → Done
└── planning-engine          vault/Needs_Action/ → Plans + Approvals
```

All processes:
- Restart automatically on crash
- Write to `logs/pm2-*.log`
- Load `.env` via `env_file`

## 6. Security Model

| Category | Approach |
|----------|----------|
| Secrets | `.env` file (`.gitignored`), loaded by PM2 and scripts at runtime |
| Social sessions | `~/.sessions/<platform>/` — Playwright persistent contexts |
| Gmail token | `watchers/.state/gmail_token.json` — OAuth2, not committed |
| Odoo | JSON-RPC with username/password — localhost only |
| Human approval gate | All external sends require file in `vault/Approved/` |
| Audit trail | Every action logged to `vault/Logs/YYYY-MM-DD.jsonl` |
| Rate limiting | Max 10 actions/hour (executor in-memory) |

## 7. Lessons Learned

### L1: Playwright bot detection requires care

X.com detects bundled Chromium with `--no-sandbox` and fails silently.
**Fix**: Use `--disable-blink-features=AutomationControlled` only.
Avoid `channel="chrome"` on Linux if Chrome is already running — causes immediate disconnect.

### L2: Social platform buttons are disabled until content is typed

Twitter's Post button and Facebook's Post button both start in `disabled` state.
The correct pattern is `wait_for_function` checking `button.disabled === false`
before clicking — not just `wait_for_selector` or `wait_for(visible)`.

### L3: Multi-step auth flows need graceful crash handling

X.com has a 3-step flow (email → username verification → password), not 2.
When the user closes the browser during Playwright auth, `TargetClosedError` is thrown.
**Fix**: Wrap the entire polling loop body in a single `try/except` so browser-close
is handled cleanly instead of crashing mid-loop.

### L4: Instagram requires an image for all posts

Instagram's web interface has no text-only composer path.
Every `send_instagram_post` approval must include an `Image:` field pointing to
a local file. The executor passes it as `--image-path`.

### L5: Early-return ordering matters in dry-run checks

The Instagram `create_post.py` originally checked `image_path.exists()` before
the `if dry_run: return` block, causing dry runs to fail even with a fake path.
**Fix**: Always check `dry_run` as the first guard, before any filesystem checks.

### L6: Ralph Wiggum hook fires during interactive sessions

The Stop hook re-injects prompts regardless of whether Claude was invoked interactively
or by cron. **Fix**: Introduce a `.autonomous_mode` flag file — hook only activates
when the flag exists (set by `scripts/autonomous.sh`, never by interactive sessions).

### L7: Vault as filesystem = durable, inspectable, portable state

Using markdown files as the communication medium between Claude, watchers, and the
human operator means:
- The human always has a readable, editable view of pending work
- No separate database needed
- Git provides full version history and diffing for free
- Obsidian renders it as a polished GUI without any frontend code

### L8: The `Failed/` folder is essential for observability

Before adding `vault/Failed/`, failed actions silently moved to `Done/` and
were never retried. Separating failed items lets the owner run `--retry-failed`
after diagnosing the root cause, without losing the audit trail.
